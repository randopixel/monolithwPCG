## Executive Summary

This research covers the specific pipeline for GENERATING modular building pieces procedurally using GeometryScript, saving them as reusable UStaticMesh assets, and placing them via HISM. This is the "how to build the kit" companion to the "should we use modular" decision (already answered YES in the modular-building-research plan).

**Key findings:**
1. GeometryScript can generate all 20+ piece types needed, using only primitives we already use (AppendBox, AppendSimpleSweptPolygon) plus Selection+Inset for openings (new but documented)
2. We already have `SaveMeshToAsset()` and `ConvertToHism()` in MonolithMesh -- the save and instancing infrastructure exists
3. UE5.7 auto-instancing (`r.MeshDrawCommands.DynamicInstancing`, on by default) batches identical mesh+material combinations automatically -- even without explicit HISM, placing regular StaticMeshActors with the same mesh gets automatic draw call merging
4. For the modular approach, HISM is still preferred over auto-instancing because it provides per-instance LOD, hierarchical culling, and explicit control
5. The piece generation is a one-time editor operation (~1-5ms per piece). Once saved as UStaticMesh assets, pieces are loaded/instanced like any other mesh
6. Total estimate for piece generation pipeline: ~30-40h (includes all piece types, variants, MCP actions, and tests)

## 1. Piece Generation Pipeline

### 1.3 The Generation Function

Each piece type is generated by a dedicated function. Pseudocode for the core pattern:

```cpp
UDynamicMesh* GenerateWallSolid(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();

    // Wall slab: 50cm wide x 270cm tall x 15cm thick
    FGeometryScriptPrimitiveOptions PrimOpts;
    PrimOpts.PolygroupMode = EGeometryScriptPrimitivePolygroupMode::PerFace;

    UGeometryScriptLibrary_MeshPrimitiveFunctions::AppendBox(
        Mesh, PrimOpts,
        FTransform::Identity,
        Params.Width,    // 50.0 (cm)
        Params.Thickness, // 15.0 (cm)
        Params.Height,   // 270.0 (cm)
        0, 0, 0,         // subdivisions
        EGeometryScriptPrimitiveOriginMode::Base  // pivot at bottom
    );

    // Assign MaterialIDs:
    // Front face = 0 (wall surface)
    // Back face = 1 (wall surface, may differ for exterior)
    // Top/Bottom/Sides = 2 (wall edge/cap)
    AssignMaterialIDsByNormal(Mesh, Params);

    // Box UV projection for tiling
    UGeometryScriptLibrary_MeshUVFunctions::SetMeshUVsFromBoxProjection(
        Mesh, 0, FTransform::Identity, FVector2D(100, 100));

    // Compute normals
    UGeometryScriptLibrary_MeshNormalsFunctions::ComputeSplitNormals(
        Mesh, FGeometryScriptSplitNormalsOptions());

    return Mesh;
}
```

### 1.4 Wall with Window Opening -- Selection+Inset Approach

The critical piece. From our GeometryScript deep dive research, Selection+Inset is 20x faster than Boolean Subtract and produces cleaner topology.

**Algorithm:**

```cpp
UDynamicMesh* GenerateWallWindow(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();

    // Step 1: Create wall slab (wider for window piece: 100cm or 2 cells)
    AppendBox(Mesh, ..., Params.Width, Params.Thickness, Params.Height);

    // Step 2: Select front-facing triangles in the window region
    // Window: 80cm wide x 120cm tall, centered, sill at 90cm
    FBox WindowBox(
        FVector(-40, -Params.Thickness, 90),   // min: centered X, full depth, sill height
        FVector(40, Params.Thickness, 210)      // max: 80cm wide, full depth, top at 210cm
    );
    FGeometryScriptMeshSelection Selection;
    UGeometryScriptLibrary_MeshSelectionFunctions::SelectMeshElementsInBox(
        Mesh, Selection, WindowBox,
        EGeometryScriptMeshSelectionType::Triangle,
        /*bInvert=*/false);

    // Step 3: Filter to only front-facing triangles (normal.Y > 0.5)
    FGeometryScriptMeshSelection NormalSelection;
    UGeometryScriptLibrary_MeshSelectionFunctions::SelectMeshElementsByNormalAngle(
        Mesh, NormalSelection,
        FVector(0, 1, 0),  // front normal
        30.0);             // angle tolerance

    // Step 4: Intersect selections (box AND front-facing)
    FGeometryScriptMeshSelection FinalSelection;
    UGeometryScriptLibrary_MeshSelectionFunctions::CombineMeshSelections(
        Selection, NormalSelection, FinalSelection,
        EGeometryScriptCombineMeshSelectionsMode::Intersection);

    // Step 5: Inset the selected faces (creates frame geometry)
    UGeometryScriptLibrary_MeshModelingFunctions::ApplyMeshInsetOutsetFaces(
        Mesh, EGeometryScriptMeshEditPolygroupMode::PerFace,
        FinalSelection,
        /*InsetDistance=*/5.0,  // 5cm frame width
        /*bRegroup=*/true);

    // Step 6: Delete the inset faces (creates the opening)
    // Re-select the now-inset faces (they're the ones inside the window area)
    SelectMeshElementsInBox(Mesh, InnerSelection, SmallerWindowBox, ...);
    UGeometryScriptLibrary_MeshBasicEditFunctions::DeleteSelectedTrianglesFromMesh(
        Mesh, InnerSelection);

    // Step 7: Assign MaterialIDs
    // Frame faces around opening = MaterialID 3 (window frame material)
    // Wall faces = MaterialID 0
    AssignMaterialIDsByRegion(Mesh, Params);

    return Mesh;
}
```

**Why this works better than Boolean:**
- No cutter mesh needed
- No wall-thickness centering bugs
- Frame geometry is created natively by the inset operation
- Clean topology -- no boolean artifacts, T-junctions, or degenerate triangles
- ~0.1-0.5ms vs 2-5ms per boolean

**Fallback if Selection+Inset has edge cases:** Boolean Subtract with a properly-centered cutter (we know exactly how thick the wall is since we built it). The single-piece context means no overlapping walls to confuse the boolean.

### 1.5 Alternative: Pre-tessellated Wall with Face Deletion

An even simpler approach for blockout quality:

```cpp
UDynamicMesh* GenerateWallWindow_Simple(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();

    // Create box with subdivisions that align to window boundaries
    // E.g., 4 subdivisions in X (for left-wall, left-frame, window, right-frame, right-wall)
    // and 3 in Z (below-sill, window, above-window)
    AppendBox(Mesh, ..., Width, Thickness, Height,
        /*SubdivisionsX=*/4, /*SubdivisionsY=*/0, /*SubdivisionsZ=*/3);

    // Delete the faces that correspond to the window opening
    // The subdivisions create a natural grid of quads
    SelectMeshElementsInBox(Mesh, Selection, WindowBox);
    SelectByNormal(Mesh, FrontNormal, NormalSel);
    Intersect(Selection, NormalSel, FinalSel);
    DeleteSelectedTrianglesFromMesh(Mesh, FinalSel);

    // Set MaterialID for frame faces (faces adjacent to opening)
    // Use polygroup-based selection
}
```

This is the simplest possible approach: subdivide the box so the window region is separate quads, then delete those quads. No inset, no boolean. The "frame" is just the wall material at the edges.

### 1.6 Trim/Detail Pieces via Sweep

For baseboard, cornice, window frame, and door frame pieces, use `AppendSimpleSweptPolygon` or `AppendSweepPolygon`:

```cpp
UDynamicMesh* GenerateBaseboard(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();

    // Cross-section profile: L-shaped baseboard
    // 8cm tall, 1.5cm thick, with a small lip at top
    TArray<FVector2D> Profile;
    Profile.Add(FVector2D(0, 0));       // bottom-back
    Profile.Add(FVector2D(1.5, 0));     // bottom-front
    Profile.Add(FVector2D(1.5, 7));     // front face up
    Profile.Add(FVector2D(1.0, 8));     // lip
    Profile.Add(FVector2D(0, 8));       // back at top

    // Sweep path: straight line along wall width
    TArray<FTransform> SweepPath;
    SweepPath.Add(FTransform(FVector(0, 0, 0)));
    SweepPath.Add(FTransform(FVector(Params.Width, 0, 0)));

    UGeometryScriptLibrary_MeshPrimitiveFunctions::AppendSweepPolygon(
        Mesh, PrimOpts, FTransform::Identity,
        Profile, SweepPath, /*bLoop=*/false);

    return Mesh;
}

UDynamicMesh* GenerateCornice(const FModularPieceParams& Params)
{
    // Similar to baseboard but with crown molding profile
    // Positioned at ceiling-wall junction
    TArray<FVector2D> Profile;
    Profile.Add(FVector2D(0, 0));
    Profile.Add(FVector2D(3, 0));       // projection from wall
    Profile.Add(FVector2D(2.5, 1));     // curve step 1
    Profile.Add(FVector2D(1.5, 2));     // curve step 2
    Profile.Add(FVector2D(0, 3));       // meets ceiling

    // Same sweep path as baseboard
    AppendSweepPolygon(Mesh, Profile, SweepPath);
    return Mesh;
}

UDynamicMesh* GenerateWindowFrame(const FModularPieceParams& Params)
{
    // Rectangular frame profile swept around window perimeter
    // Profile: 5cm wide x 3cm deep frame cross-section
    TArray<FVector2D> Profile;
    Profile.Add(FVector2D(0, 0));
    Profile.Add(FVector2D(5, 0));
    Profile.Add(FVector2D(5, 3));
    Profile.Add(FVector2D(0, 3));

    // Sweep path: rectangle around window opening (4 corners)
    // Window: 80cm wide x 120cm tall
    TArray<FTransform> FramePath;
    FramePath.Add(FTransform(FVector(-40, 0, 90)));   // bottom-left
    FramePath.Add(FTransform(FVector(40, 0, 90)));    // bottom-right
    FramePath.Add(FTransform(FVector(40, 0, 210)));   // top-right
    FramePath.Add(FTransform(FVector(-40, 0, 210)));  // top-left

    AppendSweepPolygon(Mesh, Profile, FramePath, /*bLoop=*/true);
    return Mesh;
}
```

### 1.7 Corner Pieces

Corner pieces are where modular kits get tricky. Two approaches:

**Approach A: L-shaped extrusion (recommended for blockout)**
```cpp
UDynamicMesh* GenerateCornerOutside(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();
    float T = Params.Thickness; // 15cm

    // L-shaped cross-section extruded to wall height
    TArray<FVector2D> LShape;
    LShape.Add(FVector2D(0, 0));
    LShape.Add(FVector2D(T, 0));       // along X wall
    LShape.Add(FVector2D(T, T));       // corner junction
    LShape.Add(FVector2D(0, T));       // along Y wall  -- wait, this is a square
    // For L-shape: need the two wall stubs
    LShape.Empty();
    LShape.Add(FVector2D(0, 0));       // inner corner
    LShape.Add(FVector2D(T + 25, 0));  // X-wall extends 25cm (half cell)
    LShape.Add(FVector2D(T + 25, T));  // X-wall outer face
    LShape.Add(FVector2D(T, T));       // corner point
    LShape.Add(FVector2D(T, T + 25)); // Y-wall extends 25cm (half cell)
    LShape.Add(FVector2D(0, T + 25)); // Y-wall outer face

    UGeometryScriptLibrary_MeshPrimitiveFunctions::AppendSimpleExtrudePolygon(
        Mesh, PrimOpts, FTransform::Identity,
        LShape, Params.Height,
        EGeometryScriptPrimitiveOriginMode::Base);

    return Mesh;
}
```

**Approach B: Two AppendBox calls (simpler, what we do now)**
```cpp
UDynamicMesh* GenerateCornerOutside_Simple(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();
    float T = Params.Thickness;
    float H = Params.Height;
    float HalfCell = 25.0f; // Half of 50cm cell

    // Wall stub along X axis
    AppendBox(Mesh, ..., FTransform(FVector(HalfCell/2 + T/2, 0, 0)),
        HalfCell + T, T, H);

    // Wall stub along Y axis
    AppendBox(Mesh, ..., FTransform(FVector(0, HalfCell/2 + T/2, 0)),
        T, HalfCell, H);

    // The two boxes overlap at the corner -- SelfUnion to clean
    ApplyMeshSelfUnion(Mesh);

    return Mesh;
}
```

Approach A (extruded L-shape) is cleaner geometry. Approach B is simpler code but needs a boolean union. For blockout, Approach B is fine; for art quality, Approach A is better.

### 1.8 Stair Pieces

Stairs are the most complex piece. Generated via stepped extrusion:

```cpp
UDynamicMesh* GenerateStairStraight(const FModularPieceParams& Params)
{
    UDynamicMesh* Mesh = AllocateMesh();

    float RiserH = 18.0f;   // IBC standard: 17.8cm max
    float TreadD = 28.0f;   // IBC standard: 27.9cm min
    float Width = Params.Width; // 50cm (1 cell) or 100cm (2 cells)
    float TotalRise = Params.Height; // 270cm
    int32 NumSteps = FMath::RoundToInt(TotalRise / RiserH); // 15 steps

    // Staircase profile (zigzag)
    TArray<FVector2D> StairProfile;
    StairProfile.Add(FVector2D(0, 0)); // bottom
    for (int32 i = 0; i < NumSteps; i++)
    {
        float Z = i * RiserH;
        float X = i * TreadD;
        StairProfile.Add(FVector2D(X, Z + RiserH));     // riser top
        StairProfile.Add(FVector2D(X + TreadD, Z + RiserH)); // tread end
    }
    // Close the bottom
    StairProfile.Add(FVector2D(NumSteps * TreadD, 0));

    // Extrude profile to stair width
    AppendSimpleExtrudePolygon(Mesh, StairProfile, Width);

    return Mesh;
}
```

**Stair landing:** Simple floor slab at intermediate height.

**Stair switchback:** Two straight flights + landing, with the second flight rotated 180 degrees. Generated as three AppendMesh calls combining straight stair + landing + rotated straight stair.

## 3. MaterialID Strategy

### 3.1 Per-Piece MaterialID Assignment

GeometryScript provides comprehensive MaterialID control:

- **`SetTriangleMaterialID(Mesh, TriangleID, MaterialID)`** -- set individual triangle
- **`RemapMaterialIDs(Mesh, FromID, ToID)`** -- bulk remap
- **`SetMaterialIDOnTriangles(Mesh, Selection, MaterialID)`** -- set from selection
- **`SetPolyGroupMaterialID(Mesh, GroupID, MaterialID)`** -- set by polygroup (best approach when using `PerFace` polygroup mode)
- **`GetTrianglesByMaterialID(Mesh, MaterialID) -> TriangleList`** -- query

**Recommended approach:** Generate with `PolygroupMode::PerFace` so each box face gets its own polygroup. Then assign MaterialIDs by polygroup, which maps to face direction (front/back/top/bottom/left/right).

### 3.2 Normal-Based MaterialID Assignment

For wall pieces where face orientation determines material:

```cpp
void AssignMaterialIDsByNormal(UDynamicMesh* Mesh, const FModularPieceParams& Params)
{
    // Front face (positive Y normal) = exterior material
    FGeometryScriptMeshSelection FrontSel;
    SelectMeshElementsByNormalAngle(Mesh, FrontSel, FVector(0,1,0), 30.0);
    SetMaterialIDOnTriangles(Mesh, FrontSel, 0); // MatID 0 = exterior

    // Back face (negative Y normal) = interior material
    FGeometryScriptMeshSelection BackSel;
    SelectMeshElementsByNormalAngle(Mesh, BackSel, FVector(0,-1,0), 30.0);
    SetMaterialIDOnTriangles(Mesh, BackSel, 1); // MatID 1 = interior

    // Top face = edge material
    FGeometryScriptMeshSelection TopSel;
    SelectMeshElementsByNormalAngle(Mesh, TopSel, FVector(0,0,1), 30.0);
    SetMaterialIDOnTriangles(Mesh, TopSel, 2); // MatID 2 = edge/cap

    // Remaining faces (sides, bottom) also get edge material
    // ... or use RemapMaterialIDs for cleanup
}
```

### 3.3 MaterialID Preservation Through Save Pipeline

**Critical detail:** When `SaveMeshToAsset()` converts DynamicMesh -> MeshDescription -> StaticMesh, MaterialIDs become Material Sections on the StaticMesh. Each unique MaterialID becomes a separate section with its own material slot.

This is already handled by our save pipeline. The saved UStaticMesh will have N material slots matching the N unique MaterialIDs used during generation.

When placing via HISM, materials are assigned to the HISM component's material slots, which map to the StaticMesh's sections.

### 3.4 Material Assignment at Assembly Time

```cpp
// When creating the HISM for wall pieces:
HISM->SetStaticMesh(WallSolidMesh);
HISM->SetMaterial(0, ExteriorBrickMaterial);   // MatID 0 = exterior
HISM->SetMaterial(1, InteriorDrywallMaterial); // MatID 1 = interior
HISM->SetMaterial(2, WallEdgeMaterial);        // MatID 2 = edge
```

This means a single piece type can be used with different material sets for different building styles. One `W_EXT_SOLID` mesh serves brick buildings, concrete buildings, and wooden buildings -- just change the HISM materials.

## 4. HISM Placement System

### 4.1 Architecture

```cpp
// One HISM component per unique (piece_type, material_set) combination
struct FModularBuildingHISM
{
    UPROPERTY()
    TMap<FName, UHierarchicalInstancedStaticMeshComponent*> PieceHISMs;
    // Key = "W_EXT_SOLID", "W_EXT_WINDOW", "F_TILE", etc.
};
```

For a typical building with 20 piece types, that's 20 HISM components. Each contains all instances of that piece type within the building.

### 4.2 Instance Placement

```cpp
void PlaceModularBuilding(const FBuildingGrid& Grid, const FModularPieceRegistry& Registry)
{
    AActor* BuildingActor = SpawnBuildingHostActor();

    // Create HISM for each piece type that will be used
    TMap<FName, UHierarchicalInstancedStaticMeshComponent*> HISMs;
    TMap<FName, TArray<FTransform>> PendingTransforms;

    // Walk grid edges, collect transforms per piece type
    for (const FGridEdge& Edge : Grid.GetAllEdges())
    {
        FName PieceType = SelectPieceForEdge(Edge, Grid);
        if (PieceType == NAME_None) continue;

        FTransform PieceTransform = ComputeEdgeTransform(Edge, Grid);
        PendingTransforms.FindOrAdd(PieceType).Add(PieceTransform);
    }

    // Walk grid cells for floors/ceilings
    for (const FGridCell& Cell : Grid.GetAllCells())
    {
        if (Cell.RoomID != OUTSIDE)
        {
            FTransform FloorTransform(FVector(Cell.X * 50, Cell.Y * 50, Cell.FloorZ));
            PendingTransforms.FindOrAdd("F_TILE").Add(FloorTransform);

            FTransform CeilingTransform(FVector(Cell.X * 50, Cell.Y * 50, Cell.FloorZ + 270));
            PendingTransforms.FindOrAdd("CL_TILE").Add(CeilingTransform);
        }
    }

    // Walk grid vertices for corners
    for (const FGridVertex& Vertex : Grid.GetAllVertices())
    {
        FName CornerType = ResolveCorner(Vertex, Grid);
        if (CornerType != NAME_None)
        {
            FTransform CornerTransform = ComputeCornerTransform(Vertex, Grid);
            PendingTransforms.FindOrAdd(CornerType).Add(CornerTransform);
        }
    }

    // Batch-create all HISM components and add instances
    for (const auto& Pair : PendingTransforms)
    {
        UStaticMesh* PieceMesh = Registry.GetMesh(Pair.Key);

        auto* HISM = NewObject<UHierarchicalInstancedStaticMeshComponent>(
            BuildingActor, *Pair.Key.ToString());
        HISM->SetupAttachment(BuildingActor->GetRootComponent());
        HISM->RegisterComponent();
        HISM->SetStaticMesh(PieceMesh);
        HISM->SetMobility(EComponentMobility::Static);

        // Assign materials from building style
        AssignBuildingMaterials(HISM, Pair.Key, BuildingStyle);

        // Batch add all instances at once
        HISM->AddInstances(Pair.Value, /*bReturnIndices=*/false);

        HISMs.Add(Pair.Key, HISM);
    }
}
```

### 4.3 Performance Expectations

**Per building (7 rooms, ~200 instances):**

| Metric | Current (Merged Mesh) | Modular (HISM) |
|--------|-----------------------|----------------|
| Generation time | 200-2000ms | 5-20ms |
| Draw calls | 1-3 (1 mesh, multi-material) | 10-20 (per piece type) |
| Unique GPU memory | ~50-200KB (724 tris) | ~5KB shared (piece meshes loaded once) |
| Triangle count | 724 (custom) | ~1500-3000 (more geometry from overlapping corners) |
| Collision complexity | 1 complex mesh | Per-piece simple boxes |
| LOD support | None (dynamic mesh) | Per-piece-type LOD |
| Nanite support | No | Yes |

**At city block scale (20 buildings, ~4000 instances):**

| Metric | Current | Modular |
|--------|---------|---------|
| Generation time | 4-40 seconds | 0.1-0.4 seconds |
| Draw calls | 20-60 | 10-20 (shared across buildings!) |
| Unique GPU memory | 1-4MB (20 unique meshes) | ~5KB (same pieces, instanced) |

**Key insight:** HISM draw calls are per piece TYPE, not per building. 20 buildings using the same 15 piece types = still 15 draw calls total for all wall geometry. This is the massive win over merged meshes.

### 4.4 Auto-Instancing as Fallback

UE5.7 has `r.MeshDrawCommands.DynamicInstancing` (enabled by default since 4.22). This automatically batches draw calls for identical mesh+material combinations even without explicit HISM.

**What this means:** Even if we place modular pieces as individual StaticMeshActors (no HISM), the engine will batch them. However, HISM is still preferred because:
1. Hierarchical culling (HISM culls branches of the tree)
2. Per-instance LOD transitions
3. Less CPU overhead than managing thousands of actors
4. Explicit control over batching boundaries

### 4.5 DrawCallReducer Integration

Our DrawCallReducer plugin merges draw calls for meshes with identical materials. In the modular pipeline:
- All `W_EXT_SOLID` instances share the same mesh and material
- DrawCallReducer can merge their HISM draw calls with other HISM components using the same material
- Net effect: if exterior walls, floors, and ceilings all use the same material, DrawCallReducer could merge those 3 HISM draw calls into 1

**Combined with HISM + auto-instancing:** The total draw call count for a city block could be as low as 5-10 for all structural geometry.

### 4.6 Per-Instance Custom Data

HISM supports per-instance custom data floats, accessible in materials:

```cpp
HISM->NumCustomDataFloats = 2;
HISM->SetCustomDataValue(InstanceIndex, 0, RoomID);      // float 0 = room ownership
HISM->SetCustomDataValue(InstanceIndex, 1, DecayLevel);  // float 1 = horror decay 0-1
```

**Known caveats (from web research):**
- HISM per-instance custom data has had bugs in older versions (indices not stable)
- In UE 4.27, custom data worked with ISM but had issues with HISM
- UE5.4+ appears to have fixed these issues, but test thoroughly
- For our editor-time use case (static buildings), this is less of a concern

In materials, access via `PerInstanceCustomData` node with the data index.

### 5.3 Versioning

The registry includes generation parameters. If parameters change (wall height, thickness, window dimensions), the regeneration logic can detect stale pieces:

```cpp
bool IsPieceStale(const FString& PieceID, const FModularPieceParams& CurrentParams)
{
    // Hash current params
    FString CurrentHash = HashParams(CurrentParams);
    // Compare with stored hash in registry
    FString StoredHash = Registry.GetParamHash(PieceID);
    return CurrentHash != StoredHash;
}
```

`generate_modular_pieces` with `force: true` regenerates all. Without force, only regenerates stale pieces.

## 7. Performance Comparison Deep Dive

### 7.1 Draw Call Analysis

**Auto-instancing (UE5.7 default):**
- `r.MeshDrawCommands.DynamicInstancing` = 1 (default)
- Engine batches identical (mesh + material + uniform params) draw calls
- Happens AFTER CPU occlusion cull, so occluded instances are not batched
- Reduces ~8000 individual mesh draws to ~4000 (50% reduction in tested scenarios)
- Debug: `r.MeshDrawCommands.LogDynamicInstancingStats 1`

**Explicit HISM:**
- Full hierarchical culling of instance tree
- Per-instance LOD support (ISM shares LOD across all instances)
- One draw call per HISM component (before auto-instancing merges further)
- Supports Nanite in UE5.4+

**Combined (HISM + auto-instancing + DrawCallReducer):**
- Best case for a 20-building city block: 5-10 draw calls for ALL structural geometry
- Worst case: 15-20 draw calls (still far better than 20-60 for merged meshes)

### 7.2 Memory Analysis

**Current (merged mesh per building):**
- Each building is a unique StaticMesh asset
- 724 tris at ~40 bytes/tri = ~29KB per building
- 20 buildings = 580KB of unique mesh data
- Plus index buffers, vertex buffers, material data

**Modular (shared pieces):**
- 20 piece types, average ~50-100 tris each = ~80KB total for ALL pieces
- Each building stores only instance transforms: 20 piece types x ~40 instances x 48 bytes/transform = ~38KB
- 20 buildings = 80KB (shared meshes) + 760KB (instance transforms) = 840KB
- But GPU mesh memory is SHARED: still just 80KB for piece geometry regardless of building count

**At 100 buildings:**
- Merged: 100 x 29KB = 2.9MB unique mesh data
- Modular: 80KB (piece meshes) + 3.8MB (instance data) = 3.88MB total, but only 80KB of GPU mesh memory

The modular approach scales linearly in instance data but the mesh data is constant. For large scenes, the GPU memory savings are massive.

### 7.3 Generation Time

**Current pipeline per building:**
```
AppendBox (walls):     ~1ms per wall segment
ApplyMeshBoolean:      ~5-50ms per opening
ComputeSplitNormals:   ~2ms
SetMeshUVs:            ~1ms
SaveMeshToAsset:       ~50ms
Total:                 200-2000ms per building
```

**Modular pipeline per building:**
```
Walk grid edges:       ~0.1ms (just math, no geometry)
Select piece types:    ~0.05ms (lookup table)
Compute transforms:    ~0.1ms
AddInstances (batch):  ~1-5ms (for ~200-400 instances)
Total:                 ~2-10ms per building
```

**Piece generation (one-time):**
```
Per piece type:        ~1-5ms (AppendBox/Extrude/Sweep)
Save to StaticMesh:    ~50ms per piece
20 pieces:             ~1-2 seconds total
```

## 8. Integration with Existing Systems

### 8.1 Floor Plan Generator -- No Changes

The floor plan generator (SP2) outputs:
- 2D grid of room IDs
- Room metadata (type, size, adjacency)
- Door placements (which edges have doors)
- Window placements (which edges have windows)

This is exactly the input the modular assembler needs. Zero changes to the floor plan generator.

### 8.2 Spatial Registry -- No Changes

The spatial registry (SP6) stores:
- Building hierarchy (block -> building -> floor -> room)
- Room dimensions and positions
- Adjacency graph

The modular assembler registers rooms identically to the current system. The registry doesn't care how the geometry is created.

### 8.3 Facade System -- Eliminated

The separate facade generation system (`MonolithMeshFacadeActions`) becomes unnecessary. In the modular approach:
- Exterior wall pieces ARE the facade
- Window pieces have window openings baked in
- No separate facade mesh, no overlapping walls, no alignment bugs

The facade system could be repurposed to add decorative overlays (signage, decals, trim patterns) on top of modular pieces.

### 8.4 Auto-Collision -- Simplified

Currently, auto-collision requires complex convex decomposition of the merged mesh. With modular pieces:
- Each piece is a simple box or near-box shape
- Collision is trivially a box collider per piece
- `SetStaticMeshCollisionFromMesh()` works perfectly on simple pieces
- No convex decomposition needed
- Faster runtime collision queries

### 8.5 Auto-Volumes -- No Changes

NavMesh, audio, post-process volume generation works on room bounds, not mesh topology. No changes needed.

## 10. Open Questions

### 10.1 Piece Width: 50cm or 100cm?

The existing research recommends 100cm (2-cell) as the standard wall piece width, with 50cm fillers for odd-length walls. This reduces instance count by ~50% at the cost of needing filler pieces.

**Recommendation:** Start with 50cm pieces (simpler, matches grid exactly). Optimize to 100cm if instance count becomes a problem. The piece registry system supports both widths simultaneously.

### 10.2 Multi-Story Shared Walls

For buildings with identical floor plans on each floor, the same HISM instances can be used vertically. Each floor is just a Z-offset copy of the same instance set. This is a natural optimization that HISM handles well.

### 10.3 Non-90-Degree Angles

The current system is 90-degree only. For future diagonal walls or angled buildings, we would need:
- 45-degree wall variants
- Angled corner pieces
- OR: Fall back to GeometryScript for those specific walls

**Recommendation:** 90-degree only for Phase 1. Add diagonals in a future phase if needed.

### 10.4 Level of Detail

Each piece should have LODs generated automatically:
- LOD0: Full detail (as generated)
- LOD1: Simplified (remove trim/frame details)
- LOD2: Billboard/impostor (for distant buildings)

UE5's auto-LOD tools can generate these from the saved StaticMesh assets. HISM handles per-instance LOD transitions natively.
