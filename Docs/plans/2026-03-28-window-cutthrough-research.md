# Window Boolean Operations: Full Cut-Through vs Indentation

**Date:** 2026-03-28
**Status:** Research Complete
**Related:** `2026-03-28-boolean-doors-research.md`, `2026-03-28-facade-window-research.md`, `2026-03-28-thin-wall-geometry-research.md`

---

## Summary

Windows in `generate_facade` appear as dark rectangular indentations rather than full cut-throughs. After analyzing the code and researching GeometryScript boolean behavior, there are **three distinct issues** causing this, in order of severity:

1. **The facade is a SEPARATE mesh from the building** -- even if the facade boolean cuts clean holes, the building's solid exterior walls remain behind it (the primary visual issue)
2. **Cutter Z-positioning with `Base` origin mode** -- cutters start at the sill height and extend upward, but the wall slab also uses `Base` mode with the same origin, so the cutter is positioned correctly relative to the wall (this is NOT a bug)
3. **Glass pane geometry fills the opening** -- `AddGlassPanes()` places a 1cm-thick box in every window opening, making holes appear opaque from the exterior

The root cause is **architectural**: the facade system generates its own wall slab (`BuildWallSlab`) with boolean-subtracted windows, but the `create_building_from_grid` building mesh still has solid, uncut exterior walls directly behind the facade.

---

## Detailed Analysis

### 1. Facade-Building Mesh Separation (Root Cause)

**The pipeline:**
```
create_building_from_grid  -->  Building mesh with SOLID exterior walls
                                  |
generate_facade            -->  Separate facade mesh with window HOLES
                                (placed on top of the building walls)
```

`GenerateFacade()` (line 796) creates a brand new `UDynamicMesh` and builds its own wall slabs via `BuildWallSlab()` (line 943). It cuts windows into THESE facade walls. But the original building from `create_building_from_grid` retains its solid exterior wall geometry generated by `GenerateWallGeometry()` (line 460 in BuildingActions.cpp).

**Result:** Two overlapping wall meshes. The facade has clean holes, but the building wall behind it is solid. The window openings show the building wall's surface as a dark rectangle (no light passes through).

**Evidence from code:**
- `BuildWallSlab()` at facade line 265: Creates a new wall box at the face location
- `GenerateWallGeometry()` at building line 460: Building already has walls at the same location
- These are saved as separate static mesh assets on separate actors

### 2. Boolean Operation Mechanics (Verified Working)

The boolean subtract itself is correctly implemented. Confirmed by analyzing the engine source:

**FMeshBoolean internals (MeshBoolean.h):**
- Uses **Fast Winding Number** (`Spatial/FastWinding.h`) to determine inside/outside
- `WindingThreshold = 0.5` -- a point must be >50% "inside" the cutter to be removed
- `SnapTolerance = FMathf::ZeroTolerance` -- very tight tolerance for vertex snapping
- `bCollapseDegenerateEdgesOnCut = true` -- cleans up degenerate edges automatically
- `bFillHoles = true` by default in `FGeometryScriptMeshBooleanOptions`

**Cutter dimensions in CutOpenings() (facade line 303):**
```cpp
float CutterDepth = WallThickness + 10.0f;  // 15 + 10 = 25cm for 15cm wall
```

This is correct: the cutter is 25cm deep for a 15cm wall, giving 5cm overshoot on each side. The cutter WILL fully penetrate the facade wall slab.

**Cutter positioning (facade line 323):**
```cpp
FVector WinCenter = FaceCenter + WidthAxis * Win.CenterX + Normal * (WallThickness * 0.5f);
WinCenter.Z = Face.WorldOrigin.Z + Win.SillZ;
```

The cutter is centered at `WallThickness * 0.5f` along the normal -- the middle of the wall. With `Base` origin mode, the cutter's bottom face is at `WinCenter.Z` (the sill height). This is correct.

**The boolean operation itself works fine.** The facade wall slab has clean holes. The issue is that those holes reveal the building's solid wall behind.

### 3. Glass Pane Occlusion

`AddGlassPanes()` (facade line 619) places a 1cm-thick opaque box at each window location. Even if the boolean creates a clean hole, the glass pane geometry fills it:

```cpp
PaneW = Win.Width - 2.0f;   // slightly inset from frame
PaneD = 1.0f;               // 1cm thick
// ... Height = Win.Height - 2.0f
```

The glass pane uses the default MaterialID (0 = wall material) unless assigned differently. This makes windows appear as solid rectangles. The pane sits at the face center (no normal offset), so it may also z-fight with the building wall behind.

## Root Cause Solutions

### Option A: Facade Replaces Building Exterior Walls (RECOMMENDED)

**Concept:** The building generator omits exterior wall geometry. The facade IS the exterior wall.

**Implementation:**
1. In `GenerateWallGeometry()`, skip `AppendBox` for segments where `Seg.bExterior == true`
2. The facade's `BuildWallSlab()` becomes the ONLY exterior wall geometry
3. Boolean subtracts cut clean holes through the single exterior wall
4. Interior rooms see through windows to the outside

**Pros:**
- No overlapping geometry
- Single source of truth for exterior walls
- Windows are real holes -- light and visibility pass through
- Reduced triangle count (no duplicate walls)
- Building descriptor already carries `ExteriorFaces` data, so facade knows exactly what to generate

**Cons:**
- Building mesh alone looks incomplete (no exterior walls until facade is applied)
- Requires tight coupling between building and facade generation
- If facade generation fails, building has gaping holes

**Effort:** ~4-6h
- Add `bool bOmitExteriorWalls` param to `create_building_from_grid` (default false for backwards compat)
- Modify `GenerateWallGeometry()` to skip exterior segments when flag is set
- Verify facade wall slab positioning matches building wall positioning exactly
- Test with multi-story, L-shaped, and non-rectangular buildings

### Option B: Building Cuts Its Own Window Openings (Pre-Cut)

**Concept:** The building generator receives window placement data and cuts its own openings. No separate facade wall slab needed.

**Implementation:**
1. `generate_facade` computes window/door placements but does NOT build wall slabs
2. Instead, it returns placement data (or writes to building descriptor)
3. A new action `apply_openings_to_building` takes the building mesh + placements, runs boolean subtract on the building's exterior walls
4. Facade only adds decorative elements (frames, sills, cornice, trim)

**Pros:**
- Clean single-mesh result
- Window openings are in the structural wall
- Facade is purely decorative (can be swapped without affecting structure)
- Matches real-world construction (windows are holes in walls, frames are trim)

**Cons:**
- Requires modifying the building mesh after creation (or during)
- More complex pipeline (two-pass)
- Boolean on full building mesh is slower than on individual facade slabs

**Effort:** ~8-12h

### Option C: Merged Mesh With Boolean (Simplest Fix)

**Concept:** After generating both building and facade, merge them and run self-union to clean up overlapping geometry.

**Implementation:**
1. `AppendMesh` the building mesh into the facade mesh (or vice versa)
2. Run `ApplyMeshSelfUnion` to resolve overlapping/interior faces
3. Save the merged result

**Pros:**
- Minimal code changes
- Works with existing pipeline
- Self-union handles overlapping geometry

**Cons:**
- Self-union is expensive (full mesh intersection)
- May produce unexpected topology at overlap regions
- Does not inherently create window holes in the building wall (still two wall layers, just merged)
- Self-union with `bTrimFlaps` might remove facade detail unexpectedly

**Effort:** ~2-3h but unreliable results

### Option D: Remove Building Exterior Walls at Facade Faces Only

**Concept:** After facade generation, boolean-subtract the facade wall volume from the building mesh.

**Implementation:**
1. Generate building normally (with exterior walls)
2. Generate facade normally (with its own wall slabs + window cuts)
3. Create a "removal volume" matching each facade wall slab (but without window holes)
4. Boolean-subtract the removal volumes from the building mesh
5. The facade's pre-cut wall replaces the removed section

**Pros:**
- Building mesh stays complete until facade is applied
- Selective removal (only faces with facades)
- Building without facade still looks complete

**Cons:**
- Extra boolean operation on building mesh
- Must carefully match volumes to avoid gaps
- More complex than Option A

**Effort:** ~6-8h

## Recommended Solution: Option A with Fallback

**Primary approach: Option A (facade replaces exterior walls)**

This is the cleanest architectural solution. The building descriptor already provides all the data the facade needs. The only change is suppressing exterior wall generation in the building when a facade will be applied.

**Implementation plan:**

### Phase 1: Building Generator Change (~2h)
1. Add `omit_exterior_walls` boolean param to `create_building_from_grid` (default: `false`)
2. When true, skip `AppendBox` calls for exterior wall segments in `GenerateWallGeometry()`
3. Still emit `FExteriorFaceDef` entries in the descriptor (facade needs these)
4. Interior walls, floors, and ceilings are unaffected

### Phase 2: Facade Wall Alignment Verification (~2h)
1. Verify that `BuildWallSlab()` positions match exactly where building exterior walls would be
2. Current code computes `FaceCenter` from `Face.WorldOrigin` -- same origin used by building
3. Test with: single room, multi-room, L-shape, multi-story
4. Fix any coordinate mismatches

### Phase 3: Glass Pane Material Fix (~1h)
1. Assign glass panes a dedicated MaterialID (currently uses default 0 = wall)
2. Apply translucent or masked material at spawn time
3. Or: make glass panes optional (param `add_glass_panes: false`)

### Phase 4: Interior Visibility (~1h)
1. With exterior walls removed and facade having real holes, verify:
   - Interior room walls are visible through windows
   - No backface culling issues on interior walls
   - Light enters through window openings
2. Consider `bFlipOrientation` if normals face wrong way

**Total estimate: ~6h**

## Boolean Best Practices (Reference)

For future procedural geometry work, these are the verified best practices from engine source analysis:

### Cutter Sizing
- **Minimum cutter depth:** `WallThickness + 2 * Margin` where Margin >= 1cm
- Current code uses `WallThickness + 10cm` which is generous and correct
- Too-deep cutters are fine (winding number handles cleanup)
- Too-shallow cutters cause partial cuts or indentations

### Cutter Positioning
- **Center the cutter on the wall center plane**, not the wall surface
- For `Base` origin mode: cutter's bottom face sits at the transform Z
- For `Center` origin mode: cutter's center sits at the transform position
- Current facade code correctly uses `Normal * (WallThickness * 0.5f)` offset

### Avoiding Coplanar Faces
- The 0.5cm inset (`Win.Width - 1.0f`, `Win.Height - 1.0f`) prevents cutter faces from being coplanar with wall faces
- This is correct and important -- coplanar faces confuse the winding number test
- Minimum inset: 0.5cm (current value). Could go as low as 0.1cm.

### Merged Cutter Optimization
- Appending all cutters into one tool mesh and doing a single boolean is 30-50% faster
- Current facade code already does this correctly (line 310: `AllCutters`)
- FMeshBoolean handles multiple disjoint volumes in the tool mesh

### FMeshBoolean Internals
- Uses `TFastWindingTree` for inside/outside classification
- `WindingThreshold = 0.5` is the default; triangles with winding > 0.5 are considered "inside"
- `bFillHoles = true` in GeometryScript wrapper automatically patches boundary edges via `FMinimalHoleFiller`
- The `bSuccess` return value is ignored by GeometryScript (line 93: "ignore bSuccess, as it comes back false even if we only had small errors")
- For thin walls (< 20cm), the boolean is reliable as long as cutter extends past both faces

### Winding Number Edge Cases
- Very thin geometry (< SnapTolerance ~0.001cm) can confuse winding number
- Non-manifold input meshes produce unpredictable results
- Self-intersecting input meshes should be cleaned first with `ApplyMeshSelfUnion`

## Alternative Approach: Void-First Construction (No Booleans)

For future consideration, the most performant approach avoids booleans entirely:

### Split-Wall Segments
Instead of building a solid wall slab and cutting holes:
1. Compute window positions first
2. Build wall as segments: left-of-window, between-windows, right-of-window
3. Build header (above window) and sill-wall (below window) as separate segments
4. No boolean needed -- wall is pre-constructed with voids

**Pros:**
- Zero boolean operations
- Faster generation (linear in window count vs boolean complexity)
- Clean topology with no boolean artifacts
- Easier to assign MaterialIDs per segment

**Cons:**
- More complex generation code (many small boxes)
- Corner cases with adjacent windows, doors near corners
- Harder to implement for non-rectangular windows (arches)
- More triangles than a boolean-cut wall (each segment has its own caps)

**When to use:** When generating hundreds of buildings per frame (runtime). For editor-time generation (Monolith's use case), booleans at 25-50ms per facade are perfectly acceptable.

This approach is used by CGA shape grammars (Muller et al. 2006) and CGA++ (Schwarz & Muller 2015), where the "split" operator divides a face into sub-regions. The original CGA paper generates facades by recursively splitting wall faces at window boundaries, never using boolean subtract.

## Glass Pane Specifics

The `AddGlassPanes()` function creates opaque geometry in the window opening:

```cpp
// Glass pane: very thin, sits at wall center plane
float PaneW, PaneD;
if (bNorthSouth)
{
    PaneW = Win.Width - 2.0f;   // 2cm inset from edges
    PaneD = 1.0f;               // 1cm thick
}
```

**Issues:**
1. No dedicated MaterialID assigned -- uses default (0 = wall material)
2. Pane sits at `FaceCenter` with no normal offset -- this is the wall center, which may z-fight with the building wall behind
3. 1cm thickness is correct for a glass pane, but it must use a translucent/glass material

**Fix:** Add `GlassMaterialId` assignment to the AppendBox call (the style already defines `GlassMaterialId`), and ensure the pane is offset to sit at the exterior face of the wall rather than the center.
