## How BuildId Validation Works in UE 5.7

### The check chain (source-verified)
1. `LaunchEngineLoop.cpp:6628` calls `PluginManager.CheckModuleCompatibility()`
2. `PluginManager.cpp:2979` calls `FModuleDescriptor::CheckModuleCompatibility()` for each enabled plugin
3. `ModuleDescriptor.cpp:850` checks `ModuleManager.IsModuleUpToDate(Module.Name)` for each module
4. `ModuleManager.cpp:372–386` (`IsModuleUpToDate`): calls `FindModulePaths()` and checks `ModulePathMap.Num() == 1` (exactly one DLL found). **No version/timestamp check here.**
5. `FindModulePaths()` → `FindModulePathsInDirectory()` — this is where BuildId is used:
   - Reads the engine's main `.modules` manifest to get the reference `BuildId`
   - For each plugin's `.modules` file, checks `Manifest.BuildId == BuildId.GetValue()`
   - If they don't match, the plugin's modules are **silently excluded from `ModulePathsCache`**
   - Result: `FindModulePaths` returns 0 results for those modules → `IsModuleUpToDate` returns false → module is flagged as incompatible

### The BuildId value
- Engine BuildId = `"47537391"` on UE 5.7.3 (this install)
- Source: `Build.version` → `CompatibleChangelist` field
- Same for all users who installed the same UE 5.7.x build from Epic launcher
- For formal/promoted builds: `BuildId = CompatibleChangelist.ToString()`
- For informal builds: UBT generates a GUID or recycles the existing one from the on-disk manifest

### The .modules file format
Plain JSON at `Binaries/Win64/UnrealEditor.modules`:
```json
{
  "BuildId": "47537391",
  "Modules": {
    "ModuleName": "UnrealEditor-ModuleName.dll"
  }
}
```

### bStrictEngineVersion — does NOT exist in UE 5.7
Searched the full engine source. There is no `bStrictEngineVersion` field in:
- `FPluginDescriptor` (PluginDescriptor.h)
- Any plugin loading/validation code
This field does not exist in UE 5.7. It was a thing in older UE4 docs/forums but was removed/never formalized.

### The Bridge plugin exception
`ModuleManager.cpp:1929`: There is a hardcoded exception for `/Engine/Plugins/Bridge/`:
```cpp
if ((Manifest.BuildId == BuildId.GetValue() || SearchDirectoryName.Contains(TEXT("/Engine/Plugins/Bridge/"))))
```
This is a one-off hack for Epic's Bridge plugin — not generalizable.

### How to ship precompiled plugin binaries that work on any 5.7.x install

**The answer: ship the correct `.modules` file with the right BuildId.**

On any UE 5.7.x installation from the Epic launcher, `CompatibleChangelist` (and therefore `BuildId`) is the **same value across all patch versions of 5.7** (e.g. 5.7.0, 5.7.1, 5.7.2, 5.7.3 all share `CompatibleChangelist=47537391`). Epic keeps it stable within a minor version so that plugins built against any 5.7.x are compatible.

Steps to ship:
1. Build plugin against any UE 5.7.x engine install
2. The `Binaries/Win64/UnrealEditor.modules` file generated by UBT will have the correct `BuildId`
3. Ship that `.modules` file alongside the DLLs
4. Any user on any 5.7.x install will have the same `BuildId` in their engine manifest → your plugin passes the check

**What does NOT work:**
- Manually editing the `.modules` file with a wrong or blank BuildId
- Shipping DLLs without a `.modules` file (modules won't be found at all)
- Building against a custom/source build of the engine (will have a GUID BuildId that differs per install)
